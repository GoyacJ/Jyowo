# Desktop Internationalization Design

## Scope

Add an extensible internationalization foundation to the desktop frontend.

The first release includes:

- `zh-CN`
- `en-US`
- Runtime language switching
- Persistent language preference
- Translation of user-facing static UI copy

Backend-returned conversation content, artifact names, file paths, command output, raw event payloads, logs, and user-authored content are not translated.

## Non-Goals

- No new Tauri commands
- No backend locale negotiation
- No remote translation loading
- No machine translation
- No translation of secrets, prompts, traces, or support bundle content
- No full date, number, or plural policy beyond what the first UI copy needs

## Architecture

Use `react-i18next` and `i18next`.

Create a frontend-owned module:

```text
apps/desktop/src/shared/i18n/
  i18n.ts
  locales.ts
  resources.ts
  locales/
    en-US.ts
    zh-CN.ts
```

Ownership:

- `shared/i18n` owns locale types, resources, fallback behavior, and i18next setup.
- `app/providers.tsx` wires i18n into app startup.
- `shared/local-store/ui-preferences-store.ts` persists the selected locale.
- Feature components consume translations through `useTranslation`.

Dependency direction remains:

```text
app -> routes -> features -> shared
```

`shared/i18n` must not import app, routes, or features.

Translation resource files must keep the same key shape for every locale. Use the
English resource as the canonical structure and enforce the remaining locale files
with TypeScript `satisfies` checks or a focused test that compares resource key paths.

## Locale Model

Initial locale type:

```ts
type AppLocale = 'zh-CN' | 'en-US'
```

Default locale:

```text
zh-CN
```

Fallback locale:

```text
en-US
```

Invalid stored locale values fall back to `zh-CN`.

Adding another language later should require:

- adding a locale code to `AppLocale`
- adding one resource file
- adding one Settings option

## Namespaces

Use namespaces by product area:

```text
common
shell
conversation
settings
activity
context
artifacts
memory
evals
```

This keeps feature copy close to product boundaries without making feature modules own i18n infrastructure.

## Provider Flow

Startup flow:

1. `UiPreferencesProvider` reads `ui-preferences.json`.
2. It validates `locale`.
3. It writes locale into UI state.
4. `I18nextProvider` uses the configured i18n instance.
5. Locale changes call `i18n.changeLanguage(locale)`.
6. Preference persistence writes `locale` with the other non-sensitive UI preferences.

The app may render with the default locale before preferences finish hydrating.

Store failures stay fail-open because locale is non-security UI state.

## Settings UI

Add a language setting to `SettingsPage`.

Behavior:

- show current language
- let the user choose Chinese or English
- update UI immediately
- persist the preference through the existing UI preferences store

The control should use existing shared UI primitives and semantic tokens.

## Translation Rules

Translate:

- navigation labels
- page headings
- helper text
- buttons
- empty states
- loading states
- error labels
- static status labels

Do not translate:

- user messages
- assistant messages
- backend-provided conversation titles
- artifact titles from backend
- file paths
- command names and arguments
- raw JSON keys and payloads
- support bundle content
- provider IDs and model IDs
- MCP server IDs and tool names

## Error Handling

Missing translation behavior:

- fall back to `en-US`
- then fall back to the key

Invalid locale behavior:

- fall back to `zh-CN`
- do not throw during render

Preference read or write failure:

- keep app usable
- do not block rendering
- do not expose store error bodies in UI

## Testing

Add or update tests for:

- i18n initialization and fallback behavior
- `ui-preferences-store` locale validation and persistence
- Settings language switching
- representative shell or conversation copy changing after locale switch

Run at least:

```text
pnpm check:desktop
```

Because this feature adds `react-i18next` and `i18next`, implementation changes must
update `apps/desktop/package.json` and `pnpm-lock.yaml`.

After dependency or lockfile changes, run the root gate:

```text
pnpm check
```

If `pnpm check` cannot complete, report the exact failure and the passing sub-gates.

## Security Notes

Locale is local UI state only.

It must not affect:

- permission decisions
- Tauri command access
- policy decisions
- secret handling
- trace visibility
- replay behavior
- export content

No API keys, tokens, prompts, traces, or raw payloads are added to translation resources.
